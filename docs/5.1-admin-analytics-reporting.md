Admin Analytics & Reporting
Architecture Overview
The analytics dashboard is a client-rendered admin page at src/app/bakery-manufacturing-orders/analytics/page.tsx. It performs a single authenticated fetch against one API route and renders KPI cards, Recharts visualizations, data tables, and client-side CSV exports. There is no secondary data-fetching layer — all metrics originate from one MongoDB aggregation pipeline executed in src/app/api/admin/analytics/route.ts.

Data flow:

AnalyticsPage (dateRange state)
  → GET /api/admin/analytics?startDate={ISO}&endDate={ISO}
  → verifyAdminAPI()
  → db.collection("orders").aggregate([ $match, $facet ])
  → NextResponse.json(results[0])
  → setData(json) → KPI cards + charts + tables + export handlers
API Route: GET /api/admin/analytics
Auth: verifyAdminAPI() from @/lib/auth/adminOnly. Returns { error, status } on failure before any database access.

Query parameters:

Param Client default Server fallback Normalization
startDate
dateRange.from.toISOString() (page init: subDays(now, 30))
now - 30 days
new Date(startDateParam)
endDate
dateRange.to.toISOString() (page init: now)
now
new Date(endDateParam) then setHours(23, 59, 59, 999)
The client useEffect aborts fetch if !dateRange.to. Re-fetch triggers on any dateRange change.

Base collection: orders.

Global pre-filter ($match stage):

{
  createdAt: { $gte: startDate, $lte: endDate },
  status: { $ne: "cancelled" },
}
All facet branches inherit this filter. Orders with OrderStatus.CANCELLED are excluded from every metric. Date filtering is on Order.createdAt, not deliveryInfo.deliveryDates. Unpaid, pending, and in-progress orders are included if not cancelled.

Execution model: A single $facet stage runs parallel sub-pipelines against the matched order set. The aggregation returns one document; the route extracts results[0] and serializes it as JSON.

MongoDB Aggregation Facets
kpis
Order-level rollup:

$group: {
  _id: null,
  totalRevenue: { $sum: "$totalAmount" },
  totalOrders: { $count: {} },
  totalItemsSold: { $sum: { $sum: "$items.quantity" } },
}
Post-group $project computes:

averageOrderValue: totalOrders === 0 ? 0 : totalRevenue / totalOrders
Metrics: totalRevenue, totalOrders, totalItemsSold, averageOrderValue. Revenue uses persisted Order.totalAmount (post-discount order total), not a recomputed sum of line items.

salesOverTime
Daily revenue series for the line chart:

$group: {
  _id: { $dateToString: { format: "%Y-%m-%d", date: "$createdAt" } },
  revenue: { $sum: "$totalAmount" },
  orders: { $count: {} },
}
Sorted ascending by _id (date string). Dashboard renders revenue only; orders is computed but not displayed.

sourceStats
Order-count breakdown by Order.source:

$group: { _id: "$source", count: { $sum: 1 } }
$project: { name: { $ifNull: ["$_id", "Unknown"] }, value: "$count",_id: 0 }
Sorted descending by value. Maps to donut chart in SourceChart (dataKey="value", name as label). Known color mapping: web, instagram, phone, other, walkin; unknown keys fall back to #cbd5e1.

categoryStats
Per-line-item category attribution:

$unwind: "$items"
$addFields: { categoryObjectId: { $toObjectId: "$items.categoryId" } }
$lookup → categories on _id
$unwind with preserveNullAndEmptyArrays: true
$group by $ifNull: ["$categoryInfo.name", "Custom / Other"]:
count: $sum: "$items.quantity"
revenue: $sum: { $multiply: ["$items.price", "$items.quantity"] }
Sorted descending by revenue. Rendered in CategoryChart as pie (dataKey="revenue", nameKey="_id").

flavorStats
Raw per-item rows for client-side aggregation in FlavorChart (not pre-aggregated server-side):

$unwind: "$items"
Resolves flavor ID: $ifNull: ["$items.selectedConfig.cake.flavorId", "$items.flavorId"]
$lookup → flavors (string equality match on _id)
$lookup → categories via items.categoryId
$project: flavorId, flavor, customFlavor, flavorDocName, categoryName, quantity
Each row is one line item with resolved metadata. No server-side grouping.

diameterStats
Size popularity (bar chart):

$unwind: "$items"
$group: { _id: { $toObjectId: "$items.diameterId" }, count: { $sum: "$items.quantity" } }
$lookup → diameters
$project: { name: "$diameterInfo.name", count: 1 }
Sorted descending by count. Items without a valid diameterId (custom orders, sets) are excluded — $toObjectId on null/undefined will drop or error those documents from this facet.

topProducts
Top 10 by unit volume:

$unwind: "$items"
$group by { $toObjectId: "$items.productId" }:
totalSold: $sum: "$items.quantity"
revenue: $sum: { $multiply: ["$items.price", "$items.quantity"] }
$lookup → products
$project: { name: "$productInfo.name", totalSold, revenue }
$sort: { totalSold: -1 }, $limit: 10
Requires valid items.productId. Custom/manual line items without productId are omitted.

discountStats
Promo code performance (order-level code, item-level revenue):

$match: { "discountInfo.code": { $exists: true, $ne: null } } (within facet, after global match)
$unwind: "$items"
$group by "$discountInfo.code":
totalItemsSold: $sum: "$items.quantity"
totalRevenue: $sum: { $multiply: ["$items.price", "$items.quantity"] }
Sorted descending by totalItemsSold. Attributes all line-item revenue from discounted orders to the order's discountInfo.code, not per-item discountId.

automaticDiscountStats
Automatic discount performance via per-item discountId:

$unwind: "$items"
$match: { "items.discountId": { $exists: true, $ne: null } }
$addFields: { discountObjectId: { $toObjectId: "$items.discountId" } }
$lookup → discounts
$group by "$discountDoc.name":
totalItemsSold: $sum: "$items.quantity"
totalRevenue: $sum: { $multiply: ["$items.price", "$items.quantity"] }
Sorted descending by totalItemsSold. Groups by discount rule name from the discounts collection, not promo code string.

detailedOrders (CSV — sales report)
$sort: { createdAt: -1 }
$project: { _id, createdAt, totalAmount, itemsCount: { $size: "$items" } }
soldItemsLog (CSV — production log)
Granular line-item export:

$unwind: "$items"
Cast productId and diameterId to ObjectId
$lookup → products, diameters (both preserveNullAndEmptyArrays: true)
$project:
date: "$createdAt" (order creation date, not delivery date)
productName: $ifNull: ["$productInfo.name", "Unknown Product"]
flavor: "$items.flavor"
size: $ifNull: ["$diameterInfo.name", "Unknown Size"]
price: "$items.price"
quantity: "$items.quantity"
Sorted descending by date.

detailedProductStats (computed, not rendered)
Grouped by { productId, flavor, diameterId } with product/diameter lookups. Returned in API response but not consumed by the analytics page UI. Available for future use or direct API consumers.

Dashboard Presentation
Page: src/app/bakery-manufacturing-orders/analytics/page.tsx.

Default date range: Last 30 days (subDays(new Date(), 30) → new Date()). CustomDateRangePicker in a Popover allows arbitrary range selection; onSelectRange updates { from, to } and re-triggers fetch.

Empty state gate:

const kpis = data?.kpis?.[0] || { totalRevenue: 0, totalOrders: 0, averageOrderValue: 0, totalItemsSold: 0 };
const hasData = kpis.totalOrders > 0;
If hasData === false, the dashboard renders a single empty-state card; all charts, tables, and export are suppressed. Export button is disabled={!hasData || loading}.

FlavorChart client-side processing (src/components/admin/analytics/FlavorChart.tsx):

Derives availableCategories from categoryName (missing → "Other").
Category filter pills ("All" + sorted unique categories).
Filters raw flavorStats rows by selected category.
Resolves flavor name priority: flavorDocName → flavor → customFlavor → "Unknown" (then deleted).
Aggregates quantity per flavor name.
Sorts descending, slices top 5 for pie chart (dataKey="count").
This is the only metric with significant client-side re-aggregation; all others are server-precomputed.

Revenue Calculation Semantics
Two revenue models coexist in the pipeline:

Scope Formula Used in
Order-level
$sum: "$totalAmount"
kpis, salesOverTime, sales CSV
Item-level
$sum: { $multiply: ["$items.price", "$items.quantity"] }
categoryStats, topProducts, discountStats, automaticDiscountStats, production CSV
Order.totalAmount reflects the server-authoritative post-discount total persisted at checkout. Item-level revenue uses per-unit OrderItem.price (discounted unit price when applicable) × quantity. These can diverge when order-level adjustments, delivery fees, or rounding differ from the sum of line items. KPI and trend charts use order totals; product/category/discount breakdowns use line-item math.
