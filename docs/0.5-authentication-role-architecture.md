 Authentication & Role Architecture
The system enforces a strict two-surface authentication model. The storefront and the admin dashboard operate on completely isolated session credential stacks — they share the same underlying Firebase project and firebase-admin SDK instance, but mint, store, and validate separate HTTP-only cookies that can never cross-contaminate each other's route boundaries.

Data Storage Division: Firebase vs. MongoDB
Authentication in this system is split across two services with non-overlapping responsibilities.

Firebase Authentication is the identity layer. It owns:

Cryptographic credential storage (hashed passwords, OAuth provider tokens)
Account lifecycle metadata (email verification state, provider linkage, account creation timestamps)
Short-lived ID tokens (JWT, RS256-signed, 1-hour TTL) presented after each successful sign-in
Session cookies (Firebase-proprietary format, minted server-side via adminAuth.createSessionCookie)
MongoDB (via the User interface defined in 

src/types/index.ts
) is the authorization layer. Every MongoDB user document maps to a Firebase identity via the firebaseUid foreign key, and carries:

ts
export interface User {
  _id: string;
  firebaseUid: string;      // FK → Firebase UID — the primary join key
  email: string;
  role: "customer" | "admin"; // Coarse-grained RBAC role — the authorization gate
  name?: { first?: string; last?: string; };
  phone?: string;
  addresses?: Address[];         // Saved delivery addresses
  purchasedProductIds?: string[]; // Purchase history references
}
Firebase knows who someone is. MongoDB knows what they are allowed to do. Every sensitive server action cross-references both systems.

Storefront Authentication Flow
Entry Points: /login and /register
Both 

src/app/(client)/login/page.tsx
\login\page.tsx) and 

src/app/(client)/register/page.tsx
\register\page.tsx) are functionally identical — they both route through signInWithPopup(auth, GoogleAuthProvider) using the Firebase client SDK (@/lib/firebase/client). Firebase handles the OAuth popup and returns a UserCredential. The raw idToken is extracted via userCredential.user.getIdToken() and immediately POSTed to /api/auth/sessionLogin. The distinction between "register" and "login" is UX-only — the backend handles both with a single self-healing upsert.

/api/auth/sessionLogin — The Self-Healing Upsert (Steps A → B → C)


src/app/api/auth/sessionLogin/route.ts
 executes the following pipeline on every sign-in:

Token verification — adminAuth.verifyIdToken(idToken, true) validates the RS256 signature, audience, issuer, and expiry. The checkRevoked: true flag hits Firebase's token revocation infrastructure, blocking compromised tokens.

Session cookie minting — adminAuth.createSessionCookie(idToken, { expiresIn: 5 * 24 * 60 * 60 * 1000 }) exchanges the short-lived ID token for a Firebase session cookie with a 5-day TTL. This is set on the response as:

ts
cookieStore.set("session", sessionCookie, {
  httpOnly: true,
  secure: process.env.NODE_ENV === "production",
  maxAge: expiresIn / 1000,
  path: "/",
  sameSite: "lax",
});
The sameSite: "lax" policy permits top-level cross-site navigations (e.g. Google OAuth redirects) while blocking CSRF from third-party embeds.

MongoDB upsert — three-path self-healing logic:

Step A — Primary UID lookup: usersCollection.findOne({ firebaseUid: uid }). If a document exists, the session is already established — return { action: "login" }.
Step B — Legacy email linking: If no UID match exists but the Firebase token carries an email, the system performs findOne({ email }). If a pre-existing document is found (e.g., created before Google OAuth was introduced), it backfills the firebaseUid via $set. This is a silent one-time migration — return { action: "linked" }.
Step C — First-time account creation: If neither lookup matches, a new User document is inserted with role: "customer" as the default authorization level. Return { action: "created" } with HTTP 201.
After the session cookie is set, the client calls useAuthStore.getState().fetchProfile() to hydrate the global Zustand auth state before performing a router.push("/").

Logout


/api/auth/logout
 deletes both the session and __session cookies (the latter being a Firebase Hosting SDK legacy name):

ts
cookieStore.delete("session");
cookieStore.delete("__session");
No Firebase Admin revocation call is made on logout — the cookie's TTL is the only expiry mechanism for the storefront.

Password Reset Flow
The reset flow spans two API calls and one client-side page, and deliberately hijacks Firebase's native reset link to replace the default Google-branded email template with a custom branded one via Resend.

POST /api/auth/reset-password (

source
):

Calls adminAuth.generatePasswordResetLink(email) — Firebase Admin generates the oobCode (out-of-band code) but the link is never sent directly.
The raw Firebase URL is parsed with new URL(rawLink), and the oobCode query parameter is extracted.
A clean, branded link is constructed: ${baseUrl}/reset-password?oobCode=${oobCode}. This points to the application's own route rather than Firebase's firebaseapp.com action handler.
The link is dispatched via Resend using the PasswordResetEmail React Email template.
Security hardening: If Firebase throws auth/user-not-found or auth/invalid-email, the API returns a generic success message ("If this email exists, a reset link was sent.") to prevent user enumeration.
/auth/action (

source
):

Reads mode and oobCode from the URL search params.
On mount, calls verifyPasswordResetCode(auth, oobCode) against the Firebase client SDK to validate that the code is still active and unspent. An invalid or expired code surfaces an error UI.
On form submit, calls confirmPasswordReset(auth, oobCode, newPassword) — Firebase validates the code server-side and applies the new credential hash atomically.
Admin Authentication Flow
The admin dashboard (/bakery-manufacturing-orders/*) is architecturally isolated from the storefront. It uses an exclusive credential surface, a distinct cookie, and a stricter authorization chain with zero account-linking or auto-creation logic.

Entry Point: /bakery-manufacturing-orders/login


AdminLoginPage
 uses signInWithPopup(auth, GoogleAuthProvider) with { prompt: "select_account" } to force account selection on every attempt, preventing cached session reuse. The idToken is POSTed to /api/admin/auth/sessionLogin. The response is inspected before any redirect:

HTTP 403 → renders "Access denied. This account is not authorised to access the admin portal." — notably not a redirect, preventing information leakage about the existence of accounts.
HTTP 200 → triggers a hard window.location.href = "/bakery-manufacturing-orders" rather than router.push(). This full-page navigation ensures no React state from the login context bleeds into the admin session context.
/api/admin/auth/sessionLogin — Hardened Four-Step Validation


src/app/api/admin/auth/sessionLogin/route.ts
 enforces a significantly stricter pipeline than the storefront equivalent:

Token verification — adminAuth.verifyIdToken(idToken, true) with checkRevoked: true. Failure returns a bare 401 Unauthorized with no detail.

Primary-key-only MongoDB lookup — db.collection<User>("users").findOne({ firebaseUid: decodedToken.uid }). This is an intentional architectural constraint: there is no email fallback, no account linking, and no document creation path. An unknown UID returns 403 Forbidden immediately.

Role enforcement — If the document is found, adminUser.role !== "admin" triggers a second 403 Forbidden. A valid Firebase identity with a role: "customer" MongoDB document cannot gain admin access under any circumstances.

Session cookie minting — On success, adminAuth.createSessionCookie is called identically to the storefront flow, but the cookie is stored under the key "admin_session" with sameSite: "strict" (more restrictive than the storefront's "lax"):

ts
cookieStore.set("admin_session", sessionCookie, {
  maxAge: expiresIn / 1000,
  httpOnly: true,
  secure: process.env.NODE_ENV === "production",
  sameSite: "strict",
  path: "/",
});
sameSite: "strict" blocks the cookie from being sent on any cross-origin request, including top-level navigations, which is appropriate for a dashboard with no external OAuth redirects after login.

Route Protection: verifyAdmin and verifyAdminAPI


src/lib/auth/adminOnly.ts
 exports two guards used at different layers of the stack:

verifyAdmin() — used in RSC (React Server Component) layouts. Called unconditionally in 

AdminLayout
, it wraps every admin page render (except the login page, which is detected via the x-is-login-page request header set by Next.js middleware):

ts
export async function verifyAdmin() {
  const sessionCookie = cookieStore.get("admin_session")?.value;
  if (!sessionCookie) redirect("/bakery-manufacturing-orders/login");
  const decodedToken = await adminAuth.verifySessionCookie(sessionCookie, true);
  if (decodedToken.admin === true) {
    return { user: { role: "admin", firebaseUid: decodedToken.uid } };
  }
  throw new Error("Not an admin"); // caught → redirect()
}
Note the guard reads the Firebase custom claim decodedToken.admin — not the MongoDB role field. This means the Firebase token must have the admin: true custom claim set (via adminAuth.setCustomUserClaims) in addition to the MongoDB role being "admin". This dual-check creates defense in depth: an attacker who can manipulate the MongoDB document but not Firebase custom claims still cannot pass verifyAdmin.

verifyAdminAPI() — used inside API route handlers (all routes under /api/admin/*). Identical validation logic, but returns structured error objects instead of calling redirect(), allowing API handlers to return proper HTTP error responses rather than triggering a page redirect.

Admin Logout


/api/admin/auth/logout
 overwrites the admin_session cookie with an empty value and maxAge: 0, which instructs the browser to immediately expire and delete it:

ts
cookieStore.set("admin_session", "", { maxAge: 0, path: "/", httpOnly: true, ... });
This is semantically different from cookieStore.delete() and is preferred for cross-browser reliability in forceful expiry scenarios.
