Discount & Promotions Engine
Type Definitions
The discount data model is defined in src/types/index.ts through four discriminating types and the central Discount interface.

DiscountType — union "percentage" | "fixed". Determines the arithmetic of the line-item reduction:

"percentage": savings = (lineItemOriginalTotal *value) / 100
"fixed": savings = Math.min(unitPrice, value)* quantity — the fixed discount is capped at the unit price so finalPrice cannot go negative.
DiscountTrigger — union "automatic" | "code". Determines the activation mechanism:

"automatic": applied implicitly during calculateOrderPricing and returned by public GET /api/discounts/active (filtered with trigger: "automatic" in getActiveDiscounts()).
"code": requires an explicit promoCode from the client; without a match, documents with trigger === "code" are excluded from the applicable discount pool per line item (except the exact codeDiscount match).
DiscountTargetType — union "all" | "category" | "collection" | "seasonal" | "product". Polymorphic scope that defines how the targetIds array is evaluated:

"all": all catalog line items.
"product": targetIds.includes(dbProduct._id.toString()).
"category": targetIds.includes(dbProduct.categoryId.toString()).
"collection": intersection of dbProduct.collectionIds with targetIds.
"seasonal": intersection of dbProduct.seasonalEventIds with targetIds.
Discount — the full rule document in the MongoDB discounts collection:

export interface Discount {
  _id: string | ObjectId;
  name: string;                        // Internal label; surfaced in UI and order.discountInfo.name
  code: string;                        // Required when trigger === "code"; normalized to UPPERCASE on write
  type: DiscountType;
  value: number;                       // Percentage (10) or dollars (10.00) depending on type
  trigger: DiscountTrigger;
  targetType: DiscountTargetType;
  targetIds: (string | ObjectId)[];    // ObjectIds of target entities; empty when targetType === "all"
  isActive: boolean;
  startDate: Date | string;
  endDate: Date | string;
  minOrderValue?: number;              // Minimum subtotal for code discounts (checked after aggregation)
  usageLimit?: number;                 // null = unlimited; validated in POST /api/orders before pricing
  usedCount: number;                   // Atomically incremented after a successful standard order
}
Admin CRUD UI
Page: src/app/bakery-manufacturing-orders/discounts/page.tsx (ManageDiscountsPage).

State and loading. On mount, a Promise.all runs five parallel requests:

GET /api/admin/discounts → Discount[]
GET /api/admin/products?context=admin → ProductWithCategory[] (for ProductPicker when targetType === "product")
GET /api/admin/categories, /api/admin/collections, /api/admin/seasonals → target entities for ChipCheckbox selectors
Eager loading avoids network waterfalls when switching DiscountTargetType in the form.

UI modes. Binary toggle showForm = isCreating || editingDiscount:

List mode: AdminListItem grid (1/2/3 columns). Each card renders name, value as % or $, trigger (Ticket icon for "code", Tag for "automatic"), isActive via icon color, aggregated targetIds.length, and date range (date-fns/format).
Form mode: DiscountForm with existingDiscount | null, Cancel resets editingDiscount and isCreating.
DiscountForm (src/components/admin/DiscountForm.tsx) serializes DiscountFormData = Omit<Discount, "_id" | "targetIds"> & { targetIds: string[] }:

trigger select: when "automatic", the code field is disabled; when "code", it is required and uppercased on input.
type select: "percentage" | "fixed" controls the value placeholder.
CustomDateRangePicker → startDate/endDate as yyyy-MM-dd ISO slice.
targetType select resets targetIds: [] on change.
Conditional targets: ProductPicker, ChipCheckbox grids for category/collection/seasonal; "all" shows an informational banner.
Create — POST /api/admin/discounts (src/app/api/admin/discounts/route.ts):

Guard: verifyAdminAPI().
Required: name, type, value, trigger, targetType, startDate, endDate.
If trigger === "code" and !code → 400.
Uniqueness: findOne({ code: { $regex: /^${code}$/i } }) → 409.
Insert: code.toUpperCase(), targetIds.map(id => new ObjectId(id)), usedCount: 0, minOrderValue default 0, usageLimit default null, isActive ?? false.
Read — GET /api/admin/discounts: find({}).sort({ startDate: -1 }), without ObjectId serialization (client calls .toString()).

Update — PUT /api/admin/discounts/[id] (src/app/api/admin/discounts/[id]/route.ts):

Code uniqueness with _id: { $ne: new ObjectId(id) }.
$set on all mutable fields; targetIds recast to ObjectId.
usedCount is not updated via PUT (only on create and increment on order).
Delete — DELETE /api/admin/discounts/[id]: deleteOne; deletedCount === 0 → 404.

Client-side date validation (!startDate || !endDate) blocks submit before the server.

Promo Code Validation & Lookup
A dedicated api/discounts/code/[code]/route.ts route does not exist in the repository. Promo code validation is integrated into the server-side pricing oracle, not an isolated lookup endpoint.

Public read-only endpoint for automatic discounts:

GET /api/discounts/active/route.ts → getActiveDiscounts() from src/lib/data.ts
Query: { isActive: true, trigger: "automatic", startDate: { $lte: now }, endDate: { $gte: now } }
Purpose: catalog SSR (calculateProductPrice in src/lib/discountUtils.ts), OfferBar, badge display -15% / -$10 on product cards. Does not validate promo codes.
Promo code validation — POST /api/checkout/calculate (src/app/api/checkout/calculate/route.ts):

Request body:

{ "items": CartItem[], "promoCode"?: string }
Response:

{
  "subtotal": number,
  "discountTotal": number,
  "finalTotal": number,
  "appliedDiscount": string | null,
  "appliedCode": string | null,
  "itemBreakdown": ItemBreakdown[],
  "error": "Invalid or not applicable code" | null
}
error is set when promoCode is provided but appliedCode === null after the full algorithm run.

Algorithm: calculateOrderPricing(db, items, promoCode?) (src/lib/pricing.ts):

Rehydration (anti-tamper). Client item.price values are ignored. The server loads products, flavors, and addons by IDs from the cart and reconstructs unitPrice per item (standard cake: diameter config/multiplier + flavor + inscription + addons; productType === 'set': box label matching, combo formula, flavor surcharge).

Active discount pool:

db.collection("discounts").find({
  isActive: true,
  startDate: { $lte: now },
  endDate: { $gte: now },
})
Code resolution:

const codeDiscount = promoCode
  ? activeDiscounts.find(
      (d) =>
        d.trigger === "code" &&
        d.code.toUpperCase() === promoCode.toUpperCase().trim()
    )
  : null;
Case-insensitive match. A non-existent code does not abort the request — it returns appliedCode: null, with error set at the route level.

Per-item discount election. For each CartItem:

Filter applicableDiscounts: code discounts with _id !== codeDiscount._id are excluded.
Target matching by DiscountTargetType (see above).
For each match, savings is computed per DiscountType.
Best-savings policy: a single discount with the highest savings per line item is chosen; stacking is forbidden.
isCodeUsed = true if the winner has trigger === "code".
Global minOrderValue gate (code only):

if (codeDiscount?.minOrderValue && totalSubtotal < codeDiscount.minOrderValue) {
  return { subtotal, discountTotal: 0, finalTotal: subtotal, appliedCode: null, itemBreakdown: /*stripped*/ };
}
Return: finalTotal = totalSubtotal - totalDiscount, appliedCode: isCodeUsed && promoCode ? promoCode : null, appliedDiscount: Discount | undefined.

Client consumer — OrderSummary (src/components/(client)/checkout/OrderSummary.tsx):

useEffect on [items] → POST /api/checkout/calculate with discountCode from cartStore or without a code (automatic discounts).
handleApplyCode → same endpoint with user-entered promoCode.
On success: setDiscount(discountTotal, appliedCode, appliedDiscount) in Zustand; itemPricing map for per-line strike-through.
On error: setDiscount(0, null, null), itemPricing = {}.
Displayed total = sum of itemBreakdown[].finalPrice (server-authoritative), fallback subtotal - discountTotal.
Additional validation on persist — POST /api/orders (src/app/api/orders/route.ts):

Pre-flight usageLimit: findOne({ trigger: "code", code: promoCode.toUpperCase().trim(), isActive: true }); if usedCount >= usageLimit → 400 before calculateOrderPricing.
Re-invokes calculateOrderPricing — client totalAmount is ignored.
order.totalAmount = pricingResult.finalTotal (pending) or sum(item.rowTotal) (standard).
order.discountInfo = { code?, amount: discountTotal, name: appliedDiscount?.name } when discountTotal > 0.
Per item: originalPrice, discountName, discountId from itemBreakdown.
After a standard order: $inc: { usedCount: 1 } on { _id: appliedDiscount._id } when appliedCode is present.
Cart Total Mutation Before Payment Link Generation
Web checkout pipeline (cart → payment):

CartItem[] (Zustand cartStore)
  → POST /api/checkout/calculate { items, promoCode }
  → calculateOrderPricing → { subtotal, discountTotal, finalTotal, itemBreakdown, appliedCode }
  → cartStore.setDiscount(discountTotal, appliedCode, discountName)
  → OrderSummary renders server finalTotal
  → CheckoutClientPage submits POST /api/orders { items, promoCode: discountCode, totalAmount: subtotal }
  → Server re-runs calculateOrderPricing (authoritative)
  → Order persisted with totalAmount = discounted finalTotal
The client totalAmount in the order payload is optimistic; the server overwrites with the discounted amount. promoCode comes from useCartStore().discountCode, set only after a successful appliedCode from /api/checkout/calculate.

Cart mutations invalidate the discount: addItem, removeItem, increaseQuantity, and decreaseQuantity reset discountTotal, discountCode, and discountName in cartStore; OrderSummary re-triggers calculate on items change.

Payment link generation (custom orders):

POST /api/admin/custom-orders/[id]/convert generates:

const paymentToken = randomBytes(16).toString("hex");
totalAmount: Number(agreedPrice);
This path does not call calculateOrderPricing and does not apply Discount / DiscountType / DiscountTrigger / DiscountTargetType. The order total is set by the admin via agreedPrice, not the promotions engine. The convert route now generates and persists `paymentToken` on the Order and returns it (with `newOrderId`) so the admin UI can build the Payment Hub link `/pay/[orderId]?token=[token]`.

NOTE: This manual Interac e-Transfer flow and Payment Hub is a temporary solution. It will eventually be replaced by a direct payment gateway integration (e.g., Stripe).
