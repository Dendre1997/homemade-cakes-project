Customer Journey
Catalog Discovery and Product Configuration
The storefront catalog is the entry point for standard purchases. Product, category, collection, and seasonal event pages are rendered as server components and hydrated with live catalog data from MongoDB. Before a customer configures anything, automatic promotions may already be visible: active automatic discounts are loaded at page render and evaluated per product via the catalog pricing engine, surfacing badge text such as -15% or -$10 on eligible items. This is display-only at browse time; the authoritative discount math is always recomputed server-side later at checkout.

A customer opens a product detail page and configures a line item according to product type. For a standard cake, they select a diameter (from the product’s availableDiameterConfigs, which may apply a fixed price override or a multiplier on the base structure price), a flavor (whose surcharge is added to the calculated unit price), and optional add-ons (variant-level selections from the Addon catalog). If inscriptionSettings.isAvailable is true, they may toggle cake inscription text; when provided, the inscription fee is included in the price before any discount is applied, and the inscription string becomes part of the cart item’s composite identity key. For sets and combos, configuration differs: the customer selects a quantity/box configuration (identified by label), and for combo products may additionally configure a nested cake (flavor, diameter, optional bento inscription) plus a treats breakdown. Each selected add-on’s price is carried forward as a SelectedAddon snapshot on the cart line.

When the customer adds to cart, the client computes a unit price that already reflects any catalog-level automatic discount applied on the product page. Standard cakes deduplicate in the cart by a composite id built from product ID, flavor ID, diameter ID, and inscription text—two cakes with different inscription text are separate line items. Sets and combos always create a new cart entry (no deduplication). The cart persists in the browser via Zustand (items only in localStorage); discount state is intentionally not persisted and is reset whenever the cart mutates.

Cart Review
On the cart page, the customer reviews line items with human-readable flavor and diameter names resolved from public API lookups. The page handles three display shapes: standard cakes (flavor + diameter), sets/combos (structured breakdown including combo cake inscription and per-flavor treat counts), and add-ons (name, variant, and price per decoration). The cart subtotal shown is a client-side estimate: unit price plus add-on prices, multiplied by quantity. It is not the financial source of truth; the server recalculates everything at checkout.

Any promo code previously applied is cleared when quantities change or items are added or removed, forcing re-validation at checkout if the customer modified the cart after entering a code.

Checkout: Scheduling, Discounts, and Order Placement
Checkout is a two-step flow gated by authentication (authStore / session cookie).

Step 1 — Delivery or pickup scheduling. On mount, checkout loads the capacity oracle (POST /api/availability), which returns lead time, per-category manufacturing minutes, remaining minutes per calendar day, and admin-blocked dates. The client computes total baking minutes for the cart and compares that against per-day remaining capacity. If the entire cart cannot fit on any single day, the customer is prompted to either split the order across multiple dates, submit for admin confirmation without dates (“Buy Anyway”), or return to the cart. In split mode, each physical unit becomes a UniqueCartItem with its own ID; the customer must assign every unit to a date and time slot before advancing. Unavailable dates include lead-time windows, admin-blocked days, and days with zero remaining capacity; in single-date mode, days that cannot fit the full cart are also blocked even if partially available.

Step 2 — Customer details, discounts, and payment UI. The customer enters contact and delivery/pickup information. In parallel, the OrderSummary component calls POST /api/checkout/calculate with the cart items and an optional promo code. The server ignores client-stored prices, rehydrates product/flavor/add-on data from MongoDB, and runs calculateOrderPricing: automatic discounts apply implicitly; promo codes are matched case-insensitively and validated against active date windows, target scope, per-item best-savings rules, and global minimum order value. On success, the UI shows server-authoritative line totals (including per-item strike-through when discounted) and stores discountCode in the cart store. If a code is invalid or inapplicable, an inline error is shown and discount state is cleared—without blocking the HTTP request.

The payment area on Step 2 is a placeholder only. There is no live Square (or other PSP) SDK integration, no redirect to a payment gateway, and no customer-facing card capture. The PaymentDetails.method type includes 'square' as a schema reservation, but no route populates it during web checkout. Web orders are placed without automated payment capture; financial confirmation is handled later by admin staff through the order management UI (isPaid, paymentDetails).

Submitting the order calls POST /api/orders with items, promoCode (from the validated cart store code), delivery dates, and customer info. The server re-runs calculateOrderPricing (anti-tamper), enforces promo usageLimit if applicable, persists the Order with server-authoritative totalAmount and per-item discountInfo, and increments discount usedCount on success. If the customer chose “Buy Anyway” during scheduling, the order is created with status: pending_confirmation and empty delivery dates, entering the admin confirmation queue instead of the standard new-order flow.

Post-Purchase: Confirmation and Ongoing Visibility
On successful submission, the cart is cleared and the customer is redirected to /orders/thank-you?orderId={id}. The thank-you page performs a one-time GET /api/orders/{orderId} fetch and renders a static receipt: truncated order number (last six characters of the MongoDB ID), line items, and a pricing summary that reconstructs pre-discount subtotal from order.totalAmount plus order.discountInfo.amount. There is no real-time status polling on this page—it is a snapshot receipt, not a tracking dashboard.

For authenticated customers, the platform supports an account profile and order history (session-gated per the storefront architecture), but post-checkout status progression is primarily operational: admins advance orders through the OrderStatus lifecycle (new, pending_confirmation, in-progress, ready, delivered, etc.) via the admin orders dashboard. When an order reaches delivered from the list view, the customer’s purchasedProductIds may be updated for future catalog personalization. The customer does not receive automated payment-link checkout for standard web orders; payment is confirmed manually by admin.

Alternative Path: Custom Order Intake (Non-Catalog)
Customers who need a fully bespoke cake bypass the catalog-to-cart flow and use the /custom-order six-step wizard instead. They submit an unpriced CustomOrder request (category, free-text flavor/size preferences, design notes, reference images, contact details, optional gallery deep-link pre-fill). An admin reviews the request, sets an agreedPrice, and converts it to a production Order with source: 'admin-custom' and status: awaiting_payment. Conversion stores paymentDetails.expectedMethod as cash or e-transfer and returns a placeholder payment link URL for the admin to copy and share manually—this link does not connect to a live processor and does not run the promotions engine. The converted order then enters the same admin fulfilment and status workflow as a standard checkout order.
